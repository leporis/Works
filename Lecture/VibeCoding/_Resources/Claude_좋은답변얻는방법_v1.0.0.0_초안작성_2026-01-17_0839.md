# Claude에게 꼼꼼하고 깊이 있는 답변을 얻는 방법

**VibeCoding 교육을 위한 프롬프트 작성 가이드**

**문서 버전: v1.0.0.0**

---

## 1. 핵심 원칙

**Claude의 답변 품질은 질문의 품질에 비례합니다.**

이것이 바로 "잘 아는 사용자"와 "모르는 사용자"의 차이이며, VibeCoding 교육의 핵심이기도 합니다.

---

## 2. Claude가 "대충" 답하는 경우

| 질문 유형 | Claude의 반응 | 이유 |
|-----------|---------------|------|
| "이거 어떻게 해요?" | 일반적이고 평균적인 답변 | 맥락 부족, 가장 흔한 해석으로 처리 |
| "좋은 것 같아요?" | "네, 좋습니다" 류의 동조 | 판단 기준 없음, 무난하게 응답 |
| "만들어 주세요" | 그럴듯하지만 검증 안 된 결과물 | 요구사항 불명확 |

---

## 3. Claude가 "꼼꼼하게" 답하는 경우

| 질문 유형 | Claude의 반응 | 이유 |
|-----------|---------------|------|
| "A와 B 중 어떤 것이 나은지 기준 X, Y, Z로 비교해 주세요" | 구조화된 비교 분석 | 명확한 판단 기준 제시됨 |
| "이 계획의 문제점을 찾아주세요. 칭찬하지 마세요" | 비판적 분석 | 역할과 방향 명시됨 |
| "초등 예비교사 대상, 24시간 제약, 센서 2개 이하 조건에서 실현 가능한지 검토해 주세요" | 제약조건 기반 현실적 검토 | 구체적 맥락 제공됨 |

---

## 4. 좋은 질문의 5가지 원칙

### 원칙 1: 역할을 명시하라

```
❌ 나쁜 예
"이 계획 어때요?"

✓ 좋은 예
"이 계획을 비판적으로 검토해 주세요. 
 칭찬보다는 실현 가능성의 한계를 중심으로요."
```

**효과:** 동조/칭찬 모드를 차단하고 분석 모드를 활성화

---

### 원칙 2: 판단 기준을 제시하라

```
❌ 나쁜 예
"좋은 프로젝트 추천해 주세요"

✓ 좋은 예
"난이도 ★☆☆, 센서 2개 이하, 데이터 수집 3일 이내 
 가능한 프로젝트를 추천해 주세요"
```

**효과:** 모호한 "좋은"의 정의를 구체화하여 일관된 기준으로 평가

---

### 원칙 3: 맥락을 충분히 제공하라

```
❌ 나쁜 예
"VibeCoding 수업 계획 만들어 주세요"

✓ 좋은 예
"프로그래밍 경험 없는 교대 4학년 대상, 
 12주×2시간, 노벨엔지니어링 기반, 
 아두이노+ML+웹앱 연계, 
 전원이 전 과정 경험하는 것이 목표입니다.
 이 조건에서 수업 계획을 만들어 주세요."
```

**효과:** 일반론이 아닌 구체적 상황에 맞는 맞춤형 답변 유도

---

### 원칙 4: 원하지 않는 것을 명시하라

```
❌ 나쁜 예
"분석해 주세요"

✓ 좋은 예
"분석해 주세요. 
 단, 일반론적인 이야기 말고 
 이 구체적 상황에 적용되는 것만요."
```

**효과:** 불필요한 내용을 걸러내고 핵심에 집중

---

### 원칙 5: 출력 형식을 지정하라

```
❌ 나쁜 예
"비교해 주세요"

✓ 좋은 예
"표 형식으로 비교해 주세요. 
 항목은 난이도, 소요시간, 교육적 가치, 위험요소입니다."
```

**효과:** 구조화된 정보로 비교와 판단이 용이

---

## 5. 실제 적용 사례

### 오늘 대화에서 사용된 효과적인 질문 방식

| 질문 방식 | 효과 |
|-----------|------|
| "냉정하게 답해 주세요" | 동조/칭찬 모드 차단 |
| "비판적으로 평가해 주세요" | 분석 모드 활성화 |
| 구체적 제약조건 제시 (12주, 2시간, 교대 4학년) | 맥락 기반 현실적 답변 유도 |
| 답변에 대한 재질문 | 깊이 있는 탐구 유도 |

**이것이 "잘 아는 사용자"의 전형적인 패턴입니다.**

## 6. 요약: Claude가 대충 답하는 이유와 해결책

| Claude가 대충 답하는 이유 | 해결책 |
|--------------------------|--------|
| 맥락 부족 | 구체적 상황 설명 |
| 판단 기준 없음 | 평가 기준 명시 |
| 역할 불명확 | "비판자", "검토자" 등 역할 지정 |
| 원하는 것 모호 | 원하는 것 + 원하지 않는 것 명시 |
| 출력 형식 미지정 | 표, 목록, 단계별 등 형식 요청 |

## 7. 교육적 시사점

### 학생들에게 가르칠 수 있는 것

이 원칙들은 VibeCoding 수업의 핵심 교육 내용이 됩니다:

> "여러분이 Claude에게 '만들어 줘'라고만 하면 대충 만들어 줍니다.
> 하지만 '이 조건에서, 이 목표로, 이 형식으로 만들어 줘'라고 하면
> 훨씬 좋은 결과를 얻습니다.
> 
> 이것은 단순히 AI 사용법이 아닙니다.
> **문제를 명확히 정의하는 능력**입니다.
> 이 능력은 AI가 없어도, 어떤 문제를 해결할 때든 필요합니다."

### 3단계 요청 프로토콜과의 연결

앞서 제안한 "VibeCoding 요청 전 체크리스트"가 바로 이 원칙들을 구조화한 것입니다:

| 체크리스트 항목 | 연결되는 원칙 |
|----------------|--------------|
| 문제 정의 | 맥락 제공 |
| 기대 결과 | 판단 기준 제시 |
| 검증 계획 | 원하는 것 명시 |

---

## 8. 결론

**좋은 질문을 하는 능력 = 문제를 명확히 정의하는 능력**

이것이 VibeCoding 교육에서 학생들이 배워야 할 핵심 역량입니다.

Claude는 도구일 뿐입니다. 도구를 잘 쓰려면 **내가 무엇을 원하는지 먼저 알아야** 합니다. 이 과정에서 문제해결력, 논리적 사고력, 추상화 능력이 자연스럽게 길러집니다.
