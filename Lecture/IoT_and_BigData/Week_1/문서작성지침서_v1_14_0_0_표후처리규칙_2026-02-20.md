## 1. 개요

### 1.1 목적

본 문서는 Claude와 협업하여 각종 문서(LaTeX, Word, PowerPoint 등)를 제작할 때 적용하는 기술 지침을 정리한 것입니다.

### 1.2 관련 문서

| 문서명 | 용도 |
|--------|------|
| **버전관리지침** | 버전 번호 체계, 파일명 형식, 세션 인계 |
| **문서작성지침서 (본 문서)** | 문서 형식별 기술 지침 |
| **문서관리지침** | 이미지 작업 순서, 색상 팔레트, 코드 리뷰 |

## 2. 핵심 원칙

### 🔴 절대 금지 (Top 5)

이 문서에서 가장 중요한 규칙 5가지입니다. 다른 규칙보다 **항상 우선** 적용합니다.

| # | 규칙 | 이유 |
|:-:|------|------|
| 1 | **XeLaTeX 사용 금지** → LuaLaTeX만 사용 | XeLaTeX은 CBDT 컬러 이모지를 렌더링할 수 없음 |
| 2 | **MD 없이 최종 형식 바로 작성 금지** | 내용 검토가 어렵고, 형식 변환 시 오류가 누적됨 |
| 3 | **기존 TEX 파일 직접 편집 금지** → MD에서 재생성 | 업로드 과정에서 인코딩이 손상되어 부분 수정 시 복구 불가 |
| 4 | **쪽수 채우기용 부가 정보·간격 조작 금지** | 교육 내용의 밀도가 떨어지고 학습 품질이 저하됨 |
| 5 | **CK 범위 밖 연습문제 출제 금지** | 강의계획서·루브릭과의 평가 타당도가 훼손됨 |

> 이하 본문의 규칙은 **⚠️ 강력 권장** 또는 **💡 권장** 수준입니다. 🔴와 충돌하면 🔴가 우선합니다.

### 2.1 문서버전관리지침 준수

모든 문서 작성 및 수정 시 **버전관리지침**을 반드시 준수합니다.

| 항목 | 내용 |
|------|------|
| 버전 번호 | vX.Y.Z.W 형식 (메이저.마이너.리비전.패치) |
| 파일명 | `주요내용_v버전번호_수정사항키워드_날짜_시간.확장자` |
| 필수 구성요소 | 문서 끝에 참고자료 표 + 생성/수정 이력 표 |

### 2.2 컬러 이모지 표시

**모든 문서**를 만들거나 수정할 때 **컬러 이모지가 제대로 표시**되도록 작성합니다.

| 문서 형식 | 이모지 표시 방법 |
|----------|-----------------|
| **LaTeX** | LuaLaTeX + `emoji` 패키지 + `\setemojifont{Noto Color Emoji}` |
| **Beamer** | LuaLaTeX + `emoji` 패키지 (동일) |
| **Pandoc PDF** | XeLaTeX + 이모지→컬러 유니코드 심볼 매핑 (§5.5) |
| **Markdown** | 유니코드 이모지 직접 입력 (🍎📚💡) |
| **Word/PPT** | 유니코드 이모지 직접 입력 |

> 🔴 **절대 금지**: XeLaTeX 사용 (컬러 이모지 CBDT 형식 미지원)

### 2.3 Markdown 우선 작성 원칙

**모든 문서 작성 시 먼저 Markdown(md) 문서를 만들고**, 내용이 확정된 후 다음 단계(LaTeX, Beamer, docx 등)로 진행합니다.

| 항목 | 내용 |
|------|------|
| **원칙** | 최종 형식에 관계없이, 내용 작성은 **항상 md부터** 시작 |
| **이유** | md 단계에서 내용 검토·수정이 빠르고, 형식 변환 오류를 줄일 수 있음 |
| **적용 범위** | 강의자료, 지침서, 보고서, 슬라이드 등 **모든 문서** |

**작업 흐름:**

```
[1] Markdown(md) 작성 ◀── 내용 작성·검토·확정
    │
    ▼
[2] 최종 형식으로 변환
    ├─ LaTeX/PDF (강의교재, 보고서 등)
    ├─ Beamer/PDF (강의슬라이드, 발표 자료)
    ├─ docx (Word 문서)
    └─ 기타 형식
```

> 🔴 **절대 금지**: md 없이 바로 LaTeX·Beamer·docx 등 최종 형식으로 작성하는 것 (내용 검토가 어렵고, 형식 변환 오류가 누적됨)
>
> 💡 **예외**: 기존 md 문서를 수정하는 경우, 또는 단순 형식 변환만 하는 경우는 md 단계 생략 가능

### 2.4 발표 슬라이드 요청 시

사용자가 **발표 슬라이드**, **프레젠테이션**, **PPT 형식의 LaTeX** 등을 요청하면 **Beamer 패키지**를 사용한 LaTeX 문서로 작성합니다.

### 2.5 강의자료 작성 순서

#### 2.5.1 강의자료 정의

**강의자료**란 다음 문서들을 포괄하는 용어입니다.

| 문서 | 설명 |
|------|------|
| 강의계획서 | 전체 강의 운영 계획 (주차별 주제, 평가 방법 등) |
| 강의개요 | 해당 차시의 간략한 개요 |
| 강의노트 | 강의개요 확장 (상세 내용) |
| 강의교재 | 학습 내용 (LaTeX/PDF) |
| 강의슬라이드 | 수업용 발표 자료 (Beamer/PDF) |
| 연습문제 | 주차별 퀴즈 문제지 + 정답/해설 (md) |

#### 2.5.2 강의자료 작성 요청 시

**"강의자료를 만들어줘"** 요청 시 다음을 확인합니다.

```
1. 어떤 문서를 만들지 확인
   - 특정 문서만? (예: "강의슬라이드만 만들어줘")
   - 모든 문서? (예: "전체 강의자료 만들어줘")
   
2. 사용자 지시에 따라 해당 문서 작성 진행
```

#### 2.5.3 강의자료 검토 요청 시

**"강의자료를 검토해줘"** 요청 시 다음 항목을 **꼼꼼히 검증**합니다.

| 검토 항목 | 검토 내용 |
|----------|----------|
| 지침 준수 | 문서작성지침에 따라 작성되었는가? |
| 내용 정확성 | 내용이 정확한가? 오류는 없는가? |
| 일관성 | 내용과 형식이 문서 간 일관성을 유지하는가? |
| 그림 정확성 | 그림이 내용을 정확하게 표현하는가? |
| 그림 배치 | 그림이 적절한 위치에 배치되었는가? |
| 색상/톤 | 색상 원칙과 톤 가이드를 준수하는가? |
| 이모지 표시 | 컬러 이모지가 제대로 표시되는가? |

#### 2.5.4 강의자료 작성 절차

강의자료를 체계적으로 작성하기 위해 다음 순서를 따릅니다.

**작업 시작 방법:**

> 📌 **주차와 차시를 지정**하면 해당 차시의 강의자료 작성을 본 지침에 따라 시작합니다.
> 
> 예시: "3주차 1차시 강의자료 작성해줘"

| 단계 | 작업 내용 | 산출물 |
|:----:|----------|--------|
| 0 | **강의계획서 확인** (해당 주차/차시 내용 파악) | - |
| 1 | 강의계획서를 참조하여 해당 차시 **강의개요** 작성 | 강의개요 md 문서 |
| 2 | 강의개요를 **강의노트**로 확장 | 강의노트 md 문서 |
| 3 | 강의노트를 기반으로 **강의교재** 초안 작성 | 강의교재 md 문서 (초안) |
| 4 | **문서관리지침**에 따라 **그림과 표 제작** | 이미지 파일, 표 |
| 5 | 강의교재에 그림과 표를 삽입하고 **내용 보강** | 강의교재 md 문서 (완성) |
| 6 | 문서작성지침에 따라 **강의교재 문서 최종 정리** | 강의교재 LaTeX/PDF |
| 7 | 그림과 표를 중심으로 **강의슬라이드 문서 제작** | 강의슬라이드 Beamer/PDF |
| 8 | 해당 주차 **모든 차시 완료 후** 연습문제 작성 | 연습문제 md 문서 |

> 📌 **1~7단계**는 차시별로 반복합니다. **8단계**(연습문제)는 주차의 모든 차시가 완료된 후 1회 작성합니다.

**작업 흐름도:**

```
[0] 강의계획서 확인 ◀── 🔴 가장 먼저 실행
    │
    ▼
[1] 강의개요 (md)
    │
    ▼
[2] 강의노트 (md)
    └─ 내용 확장
    │
    ▼
[3] 강의교재 초안 (md)
    │
    ├──────────────────────────┐
    ▼                          ▼
[4] 그림/표 제작               │
    (문서관리지침 적용)         │
    │                          │
    ▼                          │
[5] 강의교재 완성 ◄────────────┘
    │
    ├──────────────────────┐
    ▼                      ▼
[6] 강의교재 최종    [7] 강의슬라이드
   (LaTeX/PDF)          (Beamer/PDF)

    ※ 1~7단계를 차시별로 반복
    │
    ▼ (해당 주차 모든 차시 완료 후)
[8] 연습문제 (md)
    ├─ 문제지: 5지선다 10 + 단답형 5
    └─ 정답/해설 포함
```

**강의노트 포함 사항:**

| 구성 요소 | 내용 |
|----------|------|
| 강의 내용 확장 | 강의개요를 구체적으로 상세화 |

**강의노트 분량 기준:**

| 항목 | 기준 |
|------|------|
| 분량 | **13쪽 ~ 17쪽** |
| 형식 | Markdown (md) |
| 내용 원칙 | **순수 수업 내용만** 포함 |

> 🔴 **절대 금지**: 쪽수 채우기 위한 부가 정보 추가 (교육 내용 밀도 저하)

> ⚠️ **분량 부족 시 대응:**
> 
> 강의노트가 **13쪽 미만**이면 해당 차시의 내용이 부족하다는 의미입니다.
> 
> **조치 사항:**
> 1. **다음 차시 내용을 당겨서** 현재 차시에 포함
> 2. **강의계획서 수정** 필요 (차시별 주제 재조정)

**강의교재 작성 기준:**

| 항목 | 기준 |
|------|------|
| 분량 | **28쪽 ~ 32쪽** |
| 형식 | LaTeX (LuaLaTeX 컴파일) |
| 출력 | PDF |
| 내용 원칙 | **순수 수업 내용만** 포함 |

> 🔴 **절대 금지**: 쪽수 채우기 위한 부가 정보 추가 (교육 내용 밀도 저하)
>
> ⚠️ **퀴즈 미포함**: 연습 문제(퀴즈)는 교재에 넣지 않고 **별도 연습문제 문서**로 작성합니다.

> ⚠️ **필수**: 4단계 그림/표 제작 시 반드시 **문서관리지침**을 적용합니다.

**연습문제 작성 기준:**

| 항목 | 기준 |
|------|------|
| 단위 | **주차별** 1개 문서 (해당 주차 모든 차시 범위) |
| 형식 | Markdown (md) |
| 문항 구성 | 5지선다형 10문항 + 단답형 5문항 (총 15문항) |
| 배점 | 5지선다형 0.5점 × 10 = 5점, 단답형 1점 × 5 = 5점, 합계 10점 |
| 출제 근거 | **강의계획서 CK 목록** + **루브릭 출제 범위·배분** |
| 정답/해설 | 문제지와 **같은 문서 내** 포함 |
| 파일명 | `AI디지털리터러시_N주차_연습문제_vX.Y.Z.W_키워드_날짜_시간.md` |

> 🔴 **절대 금지**: CK 범위를 벗어나는 문항(교재 보충 내용, 심화 개념 등) 출제 (강의계획서·루브릭과의 평가 타당도 훼손)

### 2.6 강의자료 작성 스타일

#### 2.6.1 톤 & 어조

강의자료는 **가볍고 밝은 톤**으로 작성합니다.

| 항목 | 지침 |
|------|------|
| 전체 톤 | 친근하고 접근하기 쉬운 분위기 |
| 문체 | 딱딱한 학술체보다 대화체 선호 |
| 설명 방식 | 비유, 예시, 일상적 표현 적극 활용 |
| 이모지 | 적절히 활용하여 친근감 부여 |

**권장 표현 예시:**

| ❌ 피할 표현 | ✅ 권장 표현 |
|-------------|-------------|
| "~해야 한다" | "~해 보세요", "~하면 좋아요" |
| "반드시 숙지하라" | "꼭 기억해 두세요!" |
| "다음과 같이 정의된다" | "쉽게 말하면 ~예요" |

#### 2.6.2 색상 사용 원칙

강의자료에서 **빨간색 사용을 가급적 배제**합니다.

| 항목 | 지침 |
|------|------|
| 주요 강조색 | 빨간색 대신 **파란색, 초록색, 보라색** 등 사용 |
| 경고/주의 | 빨간색 대신 **주황색** 또는 **노란색** 사용 |
| 오류 표시 | 최소한으로 사용, 필요시 **진한 주황색** |
| 배경색 | **시각적 부담이 없는 가볍고 밝은 색** 중심 |
| 가독성 | **텍스트 색과 배경색을 조화**롭게 배치하여 읽기 쉽게 |

**권장 색상 팔레트 (강의자료용):**

| 용도 | 색상명 | HEX | 비고 |
|------|--------|-----|------|
| 주요 강조 | 블루 | `#1976D2` | 기본 강조색 |
| 보조 강조 | 틸 | `#00897B` | 부가 강조 |
| 포인트 | 퍼플 | `#7B1FA2` | 특별 강조 |
| 주의/팁 | 앰버 | `#FFA000` | 주의사항 |
| 성공/완료 | 그린 | `#388E3C` | 긍정적 표시 |
| 배경 (밝은) | 크림 | `#FFF3E0` | 박스 배경 |
| 배경 (회색) | 라이트그레이 | `#F5F5F5` | 코드/수식 배경 |
| 배경 (하늘) | 라이트블루 | `#E3F2FD` | 정보 박스 배경 |
| 배경 (민트) | 라이트틸 | `#E0F2F1` | 팁 박스 배경 |

**텍스트-배경 조합 권장:**

| 배경색 | 텍스트 색 | 용도 |
|--------|----------|------|
| `#FFFFFF` (흰색) | `#333333` (진한 회색) | 본문 |
| `#F5F5F5` (연한 회색) | `#333333` (진한 회색) | 코드/수식 박스 |
| `#E3F2FD` (하늘색) | `#1565C0` (진한 파랑) | 정보/안내 박스 |
| `#FFF3E0` (크림색) | `#F57F17` (진한 앰버) | 팁/주의 박스 |
| `#E0F2F1` (민트색) | `#00695C` (진한 틸) | 예시/실습 박스 |

> 💡 **참고**: 빨간색은 부정적 인상을 줄 수 있어 교육 자료에서는 최소화합니다.

#### 2.6.3 강의슬라이드 형식

강의슬라이드는 **컬러 이모지가 제대로 표시되는 Beamer(LuaLaTeX) 형식**으로 작성하며, **metropolis 테마**를 기본 적용합니다.

| 항목 | 필수 설정 |
|------|----------|
| 문서 형식 | Beamer (LaTeX) |
| 컴파일러 | **LuaLaTeX** (XeLaTeX 사용 금지) |
| **Beamer 테마** | **metropolis** (`\usetheme[progressbar=frametitle]{metropolis}`) |
| 이모지 패키지 | `emoji` + `\setemojifont{Noto Color Emoji}` |
| 화면 비율 | 16:9 권장 (`aspectratio=169`) |

> ⚠️ **필수**: 강의슬라이드 작성 시 본 지침서의 **4. Beamer 발표 슬라이드** 섹션을 참조합니다.

#### 2.6.4 강의슬라이드 그림 활용 원칙

강의슬라이드는 **텍스트 나열이 아닌, 그림 중심의 시각적 전달**을 지향합니다. 강의개요에 정의된 그림 목록을 슬라이드에 적극 반영합니다.

| 원칙 | 설명 |
|------|------|
| **그림 우선** | 해당 단계에 배정된 그림이 있으면 반드시 슬라이드에 삽입 |
| **텍스트 최소화** | 그림이 있는 슬라이드는 핵심 키워드·한 줄 설명만 배치 |
| **그림+설명 분리** | 그림 단독 슬라이드 → 바로 다음 슬라이드에서 설명 (또는 `columns` 레이아웃으로 좌우 배치) |
| **크기 확보** | 그림은 `width=0.75\textwidth` 이상으로 충분히 크게 표시 |
| **넘침 방지** | 그림 단독 슬라이드는 최대 `0.85\textwidth`, 종횡비가 높은(세로가 긴) 그림은 `0.70\textwidth` 이하 |
| **columns 제한** | 그림+텍스트 좌우 배치 시 텍스트는 **짧은 키워드·번호 목록만** 허용. description·긴 문장은 columns 금지 |
| **출처 명시** | 그림 하단에 `\tiny` 크기로 파일명 또는 출처 표기 |

> ⚠️ **넘침 사전 점검**: 그림이 슬라이드 영역을 초과하면 내용이 잘리거나 레이아웃이 깨집니다. 아래 기준을 반드시 지킵니다.
>
> | 상황 | 최대 width | 비고 |
> |------|-----------|------|
> | 그림 단독 (가로형) | `0.85\textwidth` | 하단 한 줄 캡션 가능 |
> | 그림 단독 (세로형·정사각형) | `0.70\textwidth` | 높이가 먼저 넘침 |
> | columns 좌우 배치 | 그림 열 `0.40\textwidth` 내 `width=\textwidth` | 텍스트 열은 짧은 목록만 |
> | 그림 + 긴 설명 필요 시 | **슬라이드 분리** | 그림 → 다음 슬라이드에서 설명 |

> 💡 **슬라이드 = 시각 보조 자료**. 강의노트·교재에 있는 상세 설명을 슬라이드에 그대로 옮기지 않습니다. 그림으로 보여주고, 교수자가 말로 설명하는 구조를 기본으로 합니다.

#### 2.6.5 그림 번호 부여 원칙

강의자료의 그림 번호는 다음 원칙에 따라 부여합니다.

| 원칙 | 내용 |
|------|------|
| **기준 문서** | 각 차시의 **강의교재** |
| **부여 방식** | 교재 본문에 **등장하는 순서**대로 01, 02, 03… (2자리 zero-padding) |
| **차시 간 독립** | 같은 이미지라도 다른 차시 교재에서 사용하면 **다른 번호** 가능 |
| **차시 내 통일** | 같은 차시의 개요·노트·슬라이드는 **교재 번호를 따름** |
| **파일명과 독립** | 그림 번호는 파일명의 숫자(`_08_`)와 일치하지 않아도 됨 |

**작업 흐름:**

```
[1] 강의교재에서 등장 순서 확인
    │
    ▼
[2] 그림 번호 확정 (그림 01, 그림 02, ...)
    │
    ├─▶ 개요: 교재 번호 적용
    ├─▶ 노트: 교재 번호 적용
    └─▶ 슬라이드: 교재 번호 적용
```

> ⚠️ **금지**: 파일명 번호 순서나 제작 순서로 그림 번호를 부여하는 것 (교재 읽기 흐름과 어긋나서 학생이 혼란)
>
> ⚠️ **주의**: 그림 번호를 변경할 때는 해당 차시의 **모든 문서**(교재·개요·노트·슬라이드)를 동시에 갱신합니다.

### 2.7 동일 문제 전파 검토 원칙

한 문서에서 문제가 발견되면, **같은 원인을 공유할 가능성이 있는 관련 문서를 전수 검토**합니다.

```
┌─────────────────────────────────────────────────────────────────┐
│                                                                  │
│   "한 곳에서 문제가 발견되면, 관련 문서 전체를 검토한다."       │
│                                                                  │
│   1. 문제의 원인을 파악                                         │
│   2. 같은 원인을 공유하는 문서 목록 산출                        │
│   3. 해당 문서 전수 검토 및 수정                                │
│   4. 수정 후 재검토                                             │
│                                                                  │
└─────────────────────────────────────────────────────────────────┘
```

| 문제 유형 | 전파 검토 범위 |
|----------|---------------|
| **인코딩 오류** (TEX 한글 깨짐) | 같은 방식으로 생성된 **모든 TEX 파일** |
| **용어 불일치** | 해당 차시의 개요·노트·교재·슬라이드 **전체** |
| **그림 번호 오류** | 해당 차시의 **모든 문서** + 연습문제 |
| **버전 참조 오류** | 해당 문서를 참조하는 **모든 문서** |
| **스타일/형식 위반** | 같은 세션에서 작성된 **모든 산출물** |

> ⚠️ **금지**: 문제가 발견된 문서 하나만 수정하고 완료 선언하는 것 (같은 원인의 오류가 다른 문서에 잔존)
>
> ✅ **필수**: 문제 원인을 분석하여 동일 원인을 공유하는 모든 문서를 찾아 일괄 수정

**실패 사례:**

```
[상황] 슬라이드 TEX에서 한글 깨짐 발견
[잘못된 대응] 슬라이드만 수정 시도
[올바른 대응] 같은 세션에서 생성된 교재·노트 TEX도 즉시 확인
             → 3개 모두 이중인코딩 → 3개 모두 MD에서 재생성
```

### 2.8 표 서식 원칙

표 셀에서 **짝이 있는 기호가 서로 다른 줄에 분리되지 않도록** 열 너비를 조정합니다.

| 원칙 | 내용 | 이유 |
|------|------|------|
| **짝 기호 분리 금지** | 괄호 `()`, 따옴표 `""`, 숫자+단위(`10분`) 등이 셀 내에서 다른 줄에 나뉘지 않게 | PDF 출력 시 가독성 저하 |

| **LaTeX 구현** | `\mbox{}`로 묶기, `~`(비줄바꿈 공백), `p{너비}` 차등 지정 | §3.9.2 코드 예시 참조 |

**긴 이름 약식 표기 예시:**

| 적용 전 (셀 넘침) | 적용 후 (약식) |
|-------------------|---------------|
| MakeCode시뮬레이터LED이름표시_사용설명서_v1.0.0.0 | 사용설명서 v1.0.0.0 \* |

> 표 아래 주석 예: "\* 본 표의 문서명은 약식입니다. 정식 이름: MakeCode시뮬레이터LED이름표시_{문서종류}_v버전"

> ⚠️ **적용 기준**: 셀 텍스트가 해당 열 너비의 80%를 초과하여 강제 줄바꿈이 발생하면 약식 이름을 사용합니다. 약식 이름은 **프로젝트명을 생략**하고 문서종류+버전만 표기합니다.

#### 3.11.2 SVG → PDF 변환 파이프라인

LaTeX 문서에 SVG 이미지를 삽입할 때는 **PDF로 변환**하여 `\includegraphics`로 삽입합니다.

| 단계 | 작업 | 명령어/도구 |
|:---:|------|------------|
| 1 | SVG `font-family`를 `'Noto Sans CJK KR'`로 교체 | `sed` 또는 수동 |
| 2 | SVG → PDF 변환 | `rsvg-convert -f pdf` |
| 3 | PDF 첫 페이지 렌더링 검증 | `pdftoppm -png -r 200 -f 1 -l 1` |
| 4 | 검증 이미지 목시 확인 | 아래 체크리스트 |

⚠️ **강력 권장**: `cairosvg` 대신 `rsvg-convert` 우선 사용 (한글 렌더링 품질 우수)

```bash
# 변환 명령어
rsvg-convert -f pdf input.svg -o output.pdf

# 검증 명령어
pdftoppm -png -r 200 -f 1 -l 1 output.pdf check
# → check-1.png 생성 후 육안 확인
```

#### 3.11.3 ⚠️ PDF 이미지 삽입 후 검증 체크리스트

변환된 PDF를 LaTeX 문서에 삽입한 후, **컴파일 결과물에서** 다음을 확인합니다.

| # | 검증 항목 | 확인 방법 | 흔한 문제 |
|:-:|----------|----------|----------|
| 1 | **화살표 머리 방향** | 화살표 머리(▶)가 선의 진행 방향 끝에 위치하는가 | marker-start/marker-end 혼동, 역방향 삼각형 |
| 2 | **한글 텍스트 표시** | 모든 한글이 깨짐·누락·□ 없이 정상 표시되는가 | font-family 미교체, 폰트 미설치 |
| 3 | **컬러 이모지 표시** | SVG 내 이모지가 컬러로 정상 표시되는가 | 이모지 폰트 fallback 실패 |
| 4 | **텍스트 겹침/잘림** | 텍스트가 박스·선과 겹치거나 잘리지 않는가 | 폰트 메트릭 차이로 위치 이탈 |
| 5 | **색상 재현** | 그라디언트, 투명도, 색상이 원본 SVG와 동일한가 | PDF 변환 시 opacity 처리 차이 |
| 6 | **선/도형 연결** | 선이 의도한 지점에 정확히 연결되어 있는가 | 좌표계 변환 오차 |

#### 3.11.4 화살표 방향 검증 상세

SVG에서 `marker-start`/`marker-end`를 사용할 때, **화살표 머리(삼각형)는 반드시 선의 도착점에** 위치해야 합니다.

| 속성 | 위치 | 용도 |
|------|------|------|
| `marker-end` | 선의 **끝점** (도착) | 화살표 머리 (▶) — 일반적 사용 |
| `marker-start` | 선의 **시작점** (출발) | 화살표 꼬리 또는 역방향 표시 |

```
⚠️ 흔한 실수:

  ❌ A ──▶── B   (화살표가 중간에 위치)
  ❌ A ◀────── B   (marker-end에 역방향 삼각형)
  ✅ A ───────▶ B   (marker-end에 정방향 삼각형)
```

**검증 방법**: PDF 변환 후 렌더링 이미지에서 모든 화살표의 머리가 선의 진행 방향(도착점)을 향하고 있는지 육안 확인

### 3.15 TEX 파일 수정 원칙

업로드된 TEX 파일은 **인코딩이 손상되어 있을 가능성**이 있습니다. 부분 수정 시 정상 UTF-8 텍스트와 손상된 인코딩이 혼재하여 전체 파일이 복구 불가능해집니다.

```
┌─────────────────────────────────────────────────────────────────┐
│                                                                  │
│   "기존 TEX 파일을 직접 편집하지 않는다.                        │
│    항상 MD 원본에서 TEX를 새로 생성한다."                       │
│                                                                  │
└─────────────────────────────────────────────────────────────────┘
```

| 구분 | 허용 | 금지 |
|------|------|------|
| **TEX 수정** | MD 원본 수정 → TEX 전체 신규 생성 | 기존 TEX에 `str_replace`로 부분 수정 |
| **인코딩 복구** | MD에서 새로 생성 | 이중인코딩 역변환 시도 |
| **TEX 생성** | `create_file`로 처음부터 UTF-8 작성 | 인코딩 불확실한 파일 복사 후 수정 |

**작업 흐름:**

```
[수정 요청 발생]
    │
    ├─ TEX만 있고 MD 없음?
    │   └─ TEX → 내용 추출 → MD 작성 → TEX 신규 생성
    │
    └─ MD 원본 있음?
        └─ MD 수정 → TEX 신규 생성
```

> 🔴 **절대 금지**: 업로드된 TEX 파일을 `str_replace`, `sed`, 또는 직접 편집으로 수정하는 것 (인코딩 손상 시 복구 불가)
>
> ✅ **필수**: TEX 파일 생성 후 반드시 인코딩 검증 (`file` 명령 + 한글 문자 수 카운트)

**인코딩 검증 명령어:**

```bash
# 인코딩 확인
file -bi 파일.tex

# 정상 한글 vs mojibake 패턴 카운트
python3 -c "
t = open('파일.tex', 'r', encoding='utf-8').read()
import re
normal = len(re.findall(r'[가-힣]', t))
mojibake = len(re.findall(r'[ì|í|ë][^\x00-\x7F]', t))
print(f'정상 한글: {normal}, mojibake 패턴: {mojibake}')
"
```

사용자가 **발표 슬라이드**, **프레젠테이션**, **PPT 형식** 등을 요청하면 Beamer 패키지를 사용한 LaTeX 문서로 작성합니다.

| 항목 | **필수 설정** |
|------|-------------|

| **문서 클래스** | `beamer` |
| **한글 패키지** | `kotex` |
| **이모지 패키지** | `emoji` |

### 4.2 필수 패키지 (프리앰블)

```latex
% LuaLaTeX 컴파일 필요

\documentclass[aspectratio=169]{beamer}  % 16:9 비율

% === 필수 패키지 ===
\usepackage{fontspec}
\usepackage{kotex}
\usepackage{graphicx}
\usepackage{xcolor}
\usepackage{emoji}
\usepackage{amsmath}
\usepackage{booktabs}
\usepackage{tikz}

% === 테마 설정 (metropolis) ===
\usetheme[progressbar=frametitle]{metropolis}
% progressbar 옵션: none, head, frametitle, foot

% === 폰트 설정 (한글 + 이모지) ===
% metropolis 기본 폰트(Fira Sans) 대신 한글 지원 폰트로 재지정
\setmainfont{Noto Sans CJK KR}
\setsansfont{Noto Sans CJK KR}
\setmonofont{Noto Sans Mono CJK KR}[Scale=0.9]
\setemojifont{Noto Color Emoji}

% === 색상 정의 (프로젝트 팔레트 적용) ===
\definecolor{primary}{HTML}{1976D2}
\definecolor{darkblue}{HTML}{1E3A5F}
\definecolor{lightgray}{HTML}{F5F5F5}

% === metropolis 색상 커스터마이징 ===
\setbeamercolor{frametitle}{bg=primary, fg=white}
\setbeamercolor{progress bar}{fg=primary}
\setbeamercolor{title separator}{fg=primary}
\setbeamercolor{alerted text}{fg=primary}
\setbeamercolor{example text}{fg=primary!70!black}

% === 슬라이드 번호 표시 ===
\setbeamertemplate{frame numbering}[fraction]  % 1/20 형식
```

> 💡 **참고**: metropolis는 네비게이션 심볼을 기본 제거하므로 `\setbeamertemplate{navigation symbols}{}` 설정이 불필요합니다.

## 5. Pandoc PDF 문서 (시뮬레이터 부속 문서)

### 5.1 개요

HTML 시뮬레이터 프로젝트에는 다음 6종의 부속 문서가 동반됩니다. 이 문서들은 **Markdown → Pandoc → XeLaTeX → PDF** 파이프라인으로 빌드합니다.

| 문서 | 역할 | 대상 독자 |
|------|------|-----------|
| **개발일지** | 설계 의도, 기술 결정, 구현 상세 기록 | 개발자, 교수자 |
| **퀵가이드** | 처음 사용자를 위한 1분 속성 안내 | 학생 (비전공) |
| **사용설명서** | 모든 기능의 상세 조작법 | 학생, 교수자 |
| **활용가이드** | 수업 운영 시나리오, 교수 전략 | 교수자 |
| **이론해설서** | 시뮬레이터 내부 원리의 교육적 해설 | 교수자, 심화 학습자 |
| **이미지관리** | SVG/PNG 이미지 목록, 디자인 규격 | 개발자 |

### 5.2 작업 흐름

```
[1] Markdown(md) 작성 ◀── 내용 작성·검토·확정
    │
    ▼
[2] build.py 실행
    ├─ 전처리: 이모지 교체, H1 제거, 수동번호 제거
    ├─ 이미지 삽입: 문서별 SVG/PNG 동적 배치
    ├─ Pandoc: MD → TeX (template.tex 적용)
    ├─ fix_tex(): 이미지 경로, 비번호 섹션, 컬러 심볼
    └─ XeLaTeX ×3 → PDF
```

### 5.3 빌드 환경

| 항목 | 설정 |
|------|------|
| **엔진** | XeLaTeX (TeX Live) |
| **변환 도구** | Pandoc |
| **한글 폰트** | Noto Sans CJK KR (Regular/Bold) |
| **모노 폰트** | Noto Sans Mono CJK KR |
| **SVG→PDF** | rsvg-convert (font-family를 'Noto Sans CJK KR'로 교체 후 변환) |
| **저자명** | Changmo Yang & Claude AI |

> ⚠️ **참고**: 이 파이프라인은 XeLaTeX을 사용합니다. LuaLaTeX `emoji` 패키지 대신 **이모지→컬러 유니코드 심볼 매핑** 방식으로 컬러 이모지를 구현합니다 (§5.5 참조).

### 5.5 이모지 처리

XeLaTeX은 CBDT 컬러 이모지를 직접 렌더링할 수 없으므로, **3단계 교체** 방식을 사용합니다.

```
[1] MD 전처리: 이모지 → 플레이스홀더 (예: 🔮 → XEMJ01X)
[2] Pandoc 변환: 플레이스홀더가 TeX로 그대로 전달
[3] fix_tex(): 플레이스홀더 → \textcolor{color}{심볼} 교체
```

**매핑 예시:**

| 이모지 | 색상 | 심볼 | LaTeX 출력 |
|--------|------|------|-----------|
| 🔮 | purple | ◆ | `\textcolor{purple}{◆}` |
| ⚡ | orange | ★ | `\textcolor{orange}{★}` |
| 🚀 | teal | ▶ | `\textcolor{teal}{▶}` |
| ❤️ | red | ♥ | `\textcolor{red}{♥}` |
| 1️⃣ | blue!70!black | ① | `\textcolor{blue!70!black}{①}` |

> 💡 새 이모지가 추가되면 build.py의 `EMOJI_DEFS` 리스트에 `(이모지, 플레이스홀더, 색상, 심볼)` 튜플을 추가합니다.

| 처리 | 내용 |
|------|------|
| 이미지 경로 | 상대 경로 → 절대 경로 |

| **표 헤더 가운데 정렬** | 모든 표의 첫 행(열 제목)을 가운데 정렬 |
| **표 페이지 분할 방지** | longtable → tabular + minipage 변환 |
| 코드 블록 보호 | lstlisting을 minipage로 감싸서 페이지 분리 방지 |

#### 5.8.1 표 열 폭 자동 배분 (`auto_rebalance_columns`)

Pandoc은 마크다운 열 구분자 위치로 열 폭을 결정하므로 실제 셀 내용과 무관한 비율이 배정됩니다. 후처리에서 **셀 텍스트 길이를 분석하여 열 폭을 재배분**합니다.

**알고리즘:**

1. 각 표의 모든 셀 텍스트 표시 폭 측정 (한글=2, 영문/숫자=1)
2. 열별 최대 폭 산출
3. **짧은 열** (1줄 이내, ≤22%): 자연폭 + 여유 30%로 compact 배정
4. **긴 열**: 나머지 공간을 내용 비율로 배분

**적용 대상:**

| 테이블 유형 | 처리 |
|-------------|------|
| Pandoc 비례폭 (`p{}` 열) | 셀 내용 기반 비율 재계산 |
| Pandoc 자동폭 (`clr` 열) | `p{}` 전체폭으로 변환 후 비율 배분 |

> ⚠️ **필수**: 모든 표의 너비는 문단 너비(`\columnwidth`)로 통일합니다. 자동폭(`clr`) 테이블도 `p{}` 비례폭으로 변환합니다.

#### 5.8.2 표 헤더 가운데 정렬 (`center_table_headers`)

모든 표의 첫 행(열 제목)을 가운데 정렬로 변환합니다.

| 헤더 유형 | 처리 방법 |
|----------|----------|
| minipage 헤더 | `\raggedright` → `\centering` 교체 |
| plain text 헤더 | 각 셀을 `\multicolumn{1}{c}{}` 로 감싸기 |

> ⚠️ **필수**: `\toprule` ~ `\midrule` 사이의 헤더 영역에만 적용합니다. 데이터 행의 정렬은 변경하지 않습니다.

#### 5.8.3 표 페이지 분할 방지 (`convert_worksheet_tables`)

모든 longtable을 tabular + minipage로 변환하여 표가 페이지 경계에서 분할되지 않도록 합니다.

**변환 패턴:**

```
[변환 전] \begin{longtable}[]{...} ... \end{longtable}
[변환 후] \par\needspace{4cm}\noindent
          \begin{minipage}{\linewidth}
          \begin{tabular}{...} ... \end{tabular}
          \end{minipage}\par\medskip
```

**제거 대상 (longtable 전용 명령):** `\endhead`, `\endlastfoot`, `\noalign{}`

> 💡 `\needspace{4cm}`은 남은 공간이 4cm 미만이면 표 전체를 다음 페이지로 이동시킵니다.

#### 5.8.4 후처리 적용 순서

```
[1] 이미지 경로·keepaspectratio·PNG width
[2] 비번호 섹션
[3] 이모지 컬러
[4] URL 자동 줄바꿈
[5] 표 열 폭 자동 배분 (auto_rebalance_columns)  ◀ 표 처리
[6] 코드 블록 minipage 보호
[7] 표 longtable→tabular 변환 (convert_worksheet_tables)
[8] 표 헤더 가운데 정렬 (center_table_headers)
```

> ⚠️ **순서 중요**: 열 폭 배분(5)은 longtable 상태에서 수행하고, tabular 변환(7) 후에 헤더 정렬(8)을 적용합니다.
