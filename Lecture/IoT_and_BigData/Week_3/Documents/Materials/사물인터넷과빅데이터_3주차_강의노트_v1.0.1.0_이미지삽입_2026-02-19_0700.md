## 1. 도입: 지금 어디쯤 왔을까? (5분)

### 1.1 전체 프로젝트 흐름에서 오늘의 위치

지난 2주 동안 학생들은 바이브코딩으로 자기소개 홈페이지를 만들면서 "AI에게 말로 설명하면 코드가 나온다"는 경험을 했다. 이번 주부터는 **학기 프로젝트의 본격적인 첫걸음**을 뗀다.

전체 프로젝트는 4단계로 진행된다.

```
[1단계] 기획 (3주차) ◀── 🔵 지금 여기!
   "무엇을 측정할까? 어떤 센서를 쓸까?"
        │
        ▼
[2단계] 제작 (4주차)
   "MakeCode로 데이터수집기를 만들자"
        │
        ▼
[3단계] 수집 (5~8주차)
   "실습학교에서 실제 데이터를 모으자"
        │
        ▼
[4단계] 분석 (9~13주차)
   "바이브코딩으로 데이터를 시각화하고 해석하자"
```

> 📌 **오늘 만드는 데이터 수집 계획서**가 4주차 수집기 제작의 설계도가 되고, 5~8주차 현장 수집의 가이드가 된다. 오늘의 기획이 탄탄해야 나머지 10주가 순탄하다.

## 2. 전개 1: 프로젝트 주제 탐색 (20분)

### 2.1 좋은 프로젝트 주제의 조건 (5분)

학기 프로젝트의 주제를 정하기 전에, 어떤 주제가 "좋은 주제"인지 기준을 먼저 세운다. 아무 주제나 고르면 5~8주차 현장 수집에서 막히거나, 9주차 이후 분석할 게 없는 상황이 될 수 있다.

좋은 프로젝트 주제는 4가지 조건을 만족해야 한다.

**조건 1: 센서로 측정 가능한가?**

센서는 숫자를 만들어내는 장치다. 온도, 소음, 밝기, 거리처럼 **숫자로 바꿀 수 있는 것**만 측정할 수 있다. "학생의 기분"이나 "수업의 재미"는 센서로 직접 측정할 수 없다.

**조건 2: 학교에서 수집 가능한가?**

5~8주차에 실습학교에서 데이터를 수집한다. 따라서 **학교 안에서 마이크로비트를 설치하고 운영할 수 있는 장소**여야 한다. 야외 하천이나 산속은 현실적으로 어렵다.

**조건 3: 비교·분석이 가능한가?**

데이터를 모았는데 항상 같은 값만 나오면 분석할 거리가 없다. 시간(오전 vs 오후), 장소(1층 vs 3층), 조건(창문 열림 vs 닫힘) 등 **비교할 수 있는 변인**이 있어야 흥미로운 분석이 가능하다.

**조건 4: 교육적 의미가 있는가?**

컴퓨터교육과 학생으로서 **초등학교 현장에 적용할 수 있는 주제**를 고르면 더 좋다. "교실 환경을 쾌적하게 만들려면?"처럼 학교 생활 개선에 기여하는 주제가 이상적이다.

### 2.3 "측정 가능"과 "측정 불가능" 구분 연습 (5분)

좋은 연구 질문을 만들려면 **"센서가 직접 측정할 수 있는 것"과 "사람이 주관적으로 판단하는 것"**을 구분하는 능력이 필요하다. 이 활동에서는 학생들에게 여러 질문을 보여주고 어떤 것이 센서로 측정 가능한지 판단하게 한다.

**학생 활동: 다음 질문 중 센서로 측정 가능한 것은?**

| 질문 | 측정 가능? | 이유 |
|------|:--------:|------|
| "교실이 덥다" | ❌ → ✅ 변환 가능 | "덥다"는 주관적. "온도가 28℃ 이상이다"로 바꾸면 측정 가능 |
| "교실 온도가 28℃ 이상이다" | ✅ | 온도 센서로 측정 가능 |
| "학생들이 집중하고 있다" | ❌ | 집중은 센서로 직접 측정 불가 |
| "교실 소음이 50 이하이다" | ✅ | 마이크 센서로 측정 가능 |
| "급식이 맛있다" | ❌ | 맛은 센서로 측정 불가 |
| "복도에 사람이 지나간다" | ✅ | 거리 센서로 감지 가능 |
| "수업이 재미있다" | ❌ | 재미는 센서로 측정 불가 |
| "교실 CO₂ 농도가 1000ppm을 넘는다" | ✅ | 공기질 센서로 측정 가능 |

**핵심 포인트:**

> 📌 **센서는 현재 상태만 측정한다. 미래를 예측하지 않는다.**
>
> 센서가 "온도 30℃"라고 알려주면, 그건 **지금 이 순간 30℃**라는 뜻이다. "내일도 30℃일 것이다"라는 예측은 센서가 하는 것이 아니라 **사람이 데이터를 보고 추정하는 것**이다.

> 📌 **"측정 불가능"을 "측정 가능"으로 바꾸는 방법:**
>
> 추상적 개념 → 구체적 지표로 변환
>
> - "집중" → "소음 수준 50 이하인 시간 비율"
> - "쾌적함" → "온도 22~26℃이고 습도 40~60%인 시간 비율"
> - "혼잡함" → "1분 동안 거리 센서에 감지된 횟수"

## 3. 전개 2: 센서 선택 실습 (20분)

### 3.1 마이크로비트 V2 내장 센서 복습 (3분)

2주차에 간략히 살펴본 내장 센서를 다시 정리한다. 내장 센서만으로도 충분한 프로젝트(난이도 ⭐)가 가능하다.

| 센서 | 측정 항목 | 범위 | 특징 | 적합한 주제 |
|------|----------|------|------|------------|
| 온도 센서 | 온도 (℃) | -5 ~ 50℃ | CPU 온도 기반, 실제보다 2~3℃ 높음 | 교실 온도 변화 |
| 빛 센서 | 밝기 (0~255) | 상대값 | LED를 수광 소자로 활용 | 자연광 변화 |
| 마이크 | 소리 크기 (0~255) | 상대값 | V2 전용, 절대값(dB) 아님 | 소음 패턴 |
| 가속도 센서 | 움직임 (mg) | ±2g | X/Y/Z 3축, 흔들림·기울기 | 움직임 감지 |
| 나침반 | 방위각 (°) | 0~360° | 자기장 기반 | (활용도 낮음) |

> ⚠️ **내장 온도 센서 주의**: CPU 발열 때문에 실제 기온보다 2~3℃ 높게 나올 수 있다. 정밀한 온도 측정이 필요하면 Grove SHT40 확장 센서를 사용하는 것이 좋다.

### 3.2 Grove 확장 센서 소개 (5분)

내장 센서로 부족할 때 Grove 확장 센서를 추가로 연결할 수 있다. Grove 시스템은 **I2C 통신**이라는 방식으로 마이크로비트에 연결되며, 커넥터를 꽂기만 하면 되므로 납땜이 필요 없다.

**이 강좌에서 사용 가능한 Grove 센서:**

| 센서 | 모듈명 | 측정 항목 | 정밀도 | I2C 주소 |
|------|--------|----------|--------|----------|
| 온습도 | SHT40 | 온도 ±0.2℃, 습도 ±1.8% | 매우 높음 | 0x44 |
| 공기질 | SGP30 | CO₂ (400~60,000 ppm), TVOC | 높음 | 0x58 |
| 거리 | VL53L0X | 거리 (30~2,000 mm) | 높음 | 0x29 |
| 조도 | TSL2561 | 밝기 (0.1~40,000 lux) | 높음 | 0x29/0x39 |
| 기압 | BMP280 | 기압 (300~1,100 hPa) | 높음 | 0x76/0x77 |

> 💡 **I2C란?**: 마이크로비트와 센서가 대화하는 통신 규칙이다. 각 센서에는 고유한 "주소"가 있어서 마이크로비트가 "0x44번 센서야, 온도 알려줘"라고 요청하면 센서가 값을 보내준다.

**Grove 연결 구조:**

```
마이크로비트 V2
    │
    └── Grove 쉴드 (어댑터 보드)
            │
            ├── I2C 포트 1 → SHT40 (온습도)
            ├── I2C 포트 2 → SGP30 (공기질)
            ├── I2C 포트 3 → VL53L0X (거리)
            └── I2C 포트 4 → TSL2561 (조도)
```

> 📌 I2C 포트에는 **최대 4개 센서**를 동시에 연결할 수 있다. 단, **같은 I2C 주소를 가진 센서 2개를 동시에 연결할 수는 없다.**

### 3.3 센서 선택 도구 3종 실습 (12분)

**활용 목적**: 드래그앤드롭으로 센서를 마이크로비트에 연결해보면서 어떤 조합이 가능한지 체험한다.

4. 최대 6개 센서(내장 + 외부)까지 조합을 시도한다
5. 자신의 프로젝트 주제에 맞는 센서 조합을 메모한다

**실습 예시:**

> "교실 쾌적도 측정"을 하고 싶다면?

### 4.2 수집 간격과 데이터 양 계산 (5분)

학생들이 가장 혼란스러워하는 부분이 **"몇 분마다 기록해야 하는가"**이다. 수집 간격은 측정 대상의 변화 속도에 따라 달라진다.

**수집 간격 가이드:**

| 측정 대상 | 변화 속도 | 권장 간격 | 이유 |
|----------|:--------:|:--------:|------|
| 온도·습도 | 느림 | 5분 | 온도는 급변하지 않으므로 5분이면 충분 |
| CO₂ 농도 | 느림~보통 | 5분 | 환기 효과를 보려면 5분 간격이 적절 |
| 밝기 (자연광) | 보통 | 1~5분 | 구름에 따라 변하지만 큰 추세는 느림 |
| 밝기 (조명) | 빠름 | 30초~1분 | 조명 ON/OFF 순간을 포착하려면 짧게 |
| 소음 | 빠름 | 10초~1분 | 쉬는 시간/수업 시간 전환을 포착 |
| 거리 (통행량) | 매우 빠름 | 1초 | 사람이 지나가는 순간을 놓치지 않으려면 |

**데이터 양 계산 공식:**

```
예상 데이터 개수 = (하루 수집 시간 ÷ 수집 간격) × 수집 일수

예시: 6시간 ÷ 5분 간격 × 5일
    = (360분 ÷ 5분) × 5일
    = 72개/일 × 5일
    = 360개
```

> ⚠️ **너무 짧은 간격 주의**: 1초 간격으로 6시간 수집하면 **21,600개**! 마이크로비트 Data Logger의 저장 용량(약 64KB)을 초과할 수 있다. 간격이 짧을수록 수집 기간을 줄이거나, 중간에 데이터를 다운로드해야 한다.

### B.2 3주차 시뮬레이터 활용 시나리오

#### 시나리오 1: 센서 시뮬레이터 v9 — 주제 탐색 단계

| 항목 | 내용 |
|------|------|
| **활용 시점** | 전개 2 (25~45분) |
| **목적** | 센서 조합 체험, 프로젝트에 맞는 센서 발견 |
| **활동** | 드래그앤드롭으로 센서 연결 → 측정값 확인 → 조합 메모 |
| **교수자 역할** | 화면 공유로 시연 후 학생 개별 실습 |

**활용 단계:**

```
[1] 교수자가 화면 공유 → "교실 쾌적도" 예시로 SHT40+SGP30 연결 시연 (2분)
     │
     ▼
[2] 학생 개별 실습 → 자신의 주제에 맞는 센서 드래그앤드롭 (5분)
     │
     ▼
[3] 조합 결과 메모 → 계획서 "사용 센서" 항목에 기록
```

**시뮬레이터 v9 주요 기능 활용:**

| 기능 | 교육적 활용 |
|------|-----------|
| 드래그앤드롭 센서 연결 | 센서 조합의 물리적 구조 이해 |
| 센서별 툴팁 정보 | 측정 범위, 정밀도, I2C 주소 확인 |
| 최대 6개 센서 조합 | 내장+확장 센서 동시 활용 가능성 체험 |
| 연결 상태 표시 | I2C 주소 충돌 여부 자동 확인 |

---

#### 시나리오 2: 센서 조합 카탈로그 — 최적 조합 검색

| 항목 | 내용 |
|------|------|
| **활용 시점** | 전개 2 (25~45분), 시뮬레이터와 병행 |
| **목적** | 848가지 조합에서 프로젝트 적합 조합 필터링 |
| **활동** | 측정 항목 필터 → 난이도 필터 → 최적 조합 선택 |
| **교수자 역할** | 필터 사용법 간단 시연 |

**활용 예시:**

> 학생: "소음이랑 밝기를 동시에 측정하고 싶어요"
> → 카탈로그에서 "소리" + "밝기" 필터 적용
> → 결과: "내장 마이크 + 내장 빛 센서" (난이도 ⭐) 또는 "내장 마이크 + TSL2561" (난이도 ⭐⭐)
> → 학생이 정밀도 필요 여부에 따라 선택

---

#### 시나리오 3: 센서 검색 도구 — 키워드 기반 추천

| 항목 | 내용 |
|------|------|
| **활용 시점** | 전개 2 (25~45분), 주제를 정했지만 센서를 못 고른 학생용 |
| **목적** | 자연어 키워드로 센서 추천 받기 |
| **활동** | 프로젝트 관련 키워드 입력 → 추천 센서 확인 → 시뮬레이터에서 검증 |
| **교수자 역할** | "어떤 키워드를 넣으면 좋을지" 힌트 제공 |

**도구 간 연계 흐름:**

```
[검색 도구] 키워드 입력 → 센서 추천 목록 확인
     │
     ▼
[카탈로그] 추천된 센서 조합의 상세 스펙 확인
     │
     ▼
[시뮬레이터] 해당 센서를 드래그앤드롭으로 연결 체험
     │
     ▼
[계획서] 최종 선택한 센서를 "사용 센서" 항목에 기록
```
