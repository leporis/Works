## 1. 핵심 원칙

### 1.1 절대 원칙

```
┌─────────────────────────────────────────────────────────────────┐
│                                                                  │
│   "모든 작업물을 처음부터 끝까지 검토하여 문제점을 찾고         │
│    세밀하게 분석할 것. 무의식적 반응 금지."                      │
│                                                                  │
│   이 원칙은 코드, 문서, 이미지 등 모든 산출물에 적용한다.       │
│                                                                  │
└─────────────────────────────────────────────────────────────────┘
```

이 원칙은 코드 리뷰에만 해당하는 것이 아닙니다. Claude가 수행하는 **모든 작업**에 적용됩니다.

| 적용 대상 | 적용 방법 |
|----------|----------|
| **코드 작성/수정** | 전체 코드를 줄 단위로 읽고 검증 |
| **문서 작성/수정** | 목차-내용 일치, 버전 번호, 참조 확인 |
| **이미지 제작/수정** | 좌표, 색상, 텍스트 정합성 확인 |
| **버전 관리** | 모든 버전 표시 위치 일괄 변경 확인 |

### 1.2 금지 사항

| # | 금지 행위 | 올바른 행동 |
|:-:|----------|------------|
| 1 | 대충 훑어보고 "문제없어 보입니다" | 전체를 처음부터 끝까지 검토 |
| 2 | 일부만 보고 판단 | 모든 파일, 모든 섹션 검토 |
| 3 | 추측으로 답변 | 실제 내용에서 근거 찾기 |
| 4 | 수정 후 바로 완료 선언 | 수정 후 재검토 반복 |
| 5 | 영향 분석 생략 | 모든 호출처/사용처/참조 확인 |
| 6 | 사소한 문제를 "나중에" 미루기 | 발견 즉시 수정 |

### 1.3 교차검증 원칙

```
┌─────────────────────────────────────────────────────────────────┐
│                                                                  │
│   "수정할 부분이 없을 때까지 묻지 말고 수정한다."               │
│                                                                  │
│   1. 수정 후 전체 재검토                                        │
│   2. 추가 문제 발견 → 즉시 수정                                 │
│   3. 더 이상 발견 없을 때까지 반복                               │
│   4. 반복 후에도 없으면 그때 완료 보고                           │
│                                                                  │
└─────────────────────────────────────────────────────────────────┘
```

| 대상 | 교차검증 항목 |
|------|-------------|
| 문서 수정 | 목차 ↔ 본문 제목, 버전 번호 전체, 참조 문서 버전 |
| 코드 수정 | API 엔드포인트 ↔ 프론트엔드 호출, HTML ID ↔ JS 참조 |
| 이미지 수정 | SVG 좌표 ↔ 실제 렌더링, 색상 ↔ 문서 테마 |

### 1.4 마인드셋

```
❌ "이 정도면 됐겠지"
❌ "별 문제 없어 보이는데"
❌ "이전에 잘 됐으니까"
❌ "사소한 건 괜찮겠지"

✅ "아직 놓친 문제가 있을 수 있다"
✅ "한 줄 한 줄 확인하자"
✅ "수정이 다른 곳에 영향을 줄 수 있다"
✅ "발견한 문제는 즉시 수정하자"
```

## 2. 문서 수정 완료 체크리스트

Claude가 문서를 수정한 후, 완료를 선언하기 전에 반드시 다음을 확인합니다.

### 2.1 필수 검증 항목

| # | 항목 | 확인 내용 |
|:-:|------|----------|
| 1 | **요청 반영 완료** | 사용자가 요청한 수정 사항이 빠짐없이 모두 반영되었는가? |
| 2 | **버전 번호 일괄 변경** | 문서 내 모든 버전 표시 위치(제목, 헤더, 이력표, 파일명)가 새 버전으로 변경되었는가? |
| 3 | **목차 ↔ 본문 일치** | 목차의 제목과 본문의 제목이 정확히 일치하는가? 장 번호가 연속적인가? |
| 4 | **참조 버전 확인** | 참고자료 표의 다른 문서 버전이 최신인가? |
| 5 | **수정 이력 추가** | 생성/수정 이력 표에 이번 변경이 기록되었는가? |
| 6 | **내부 참조 정합성** | "N장 참조" 등 문서 내 교차 참조가 정확한가? |
| 7 | **파일명 갱신** | 파일명에 새 버전, 수정키워드, 날짜/시간이 반영되었는가? |

### 2.2 검증 절차

```
┌─────────────────────────────────────────────────────────────────┐
│                                                                  │
│  [수정 완료]                                                    │
│      ↓                                                          │
│  [2.1 체크리스트 검증]                                          │
│      ↓                                                          │
│  미비 사항 발견? ── Yes → 즉시 수정 → 다시 체크리스트           │
│      │                                                          │
│      No                                                         │
│      ↓                                                          │
│  [완료 보고]                                                    │
│                                                                  │
└─────────────────────────────────────────────────────────────────┘
```

### 2.3 Claude의 흔한 실수 패턴

| # | 실수 패턴 | 예방 방법 |
|:-:|----------|----------|
| 1 | 요청한 수정 중 일부를 빠뜨림 | 요청 사항을 목록화하고 하나씩 확인 |
| 2 | 문서 제목의 버전은 바꾸고 이력표는 안 바꿈 | 모든 버전 표시 위치를 grep으로 확인 |
| 3 | 참고자료에서 참조하는 문서 버전이 오래됨 | 참고자료 표의 버전을 최신과 대조 |
| 4 | 장을 추가/삭제한 후 목차를 안 고침 | 목차와 본문 제목을 1:1 대조 |
| 5 | "N장 참조" 번호가 틀림 | 문서 내 모든 "장", "절" 참조를 검색 |

## 3. 이미지 제작 프로세스

### 3.1 적용 범위

| 구분 | 대상 문서 |
|------|----------|
| 시뮬레이터 문서 | 개발일지, 퀵가이드, 사용설명서 |
| 교육 자료 | 이론강의서(이론해설서), 활용가이드, 수업지도안, 수업자료 |
| 강의 문서 | 강의계획서, 강의자료, 강의노트, 강의슬라이드 |
| 기타 | Claude와 협업하여 제작/수정하는 모든 문서 |

| 이미지 형식 | 용도 |
|------------|------|
| SVG (권장) | 다이어그램, 개념도, UI 설명, 플로우차트 |
| PNG | 스크린샷, 실제 화면 캡처 |

### 3.2 8단계 프로세스

```
┌─────────────────────────────────────────────────────────────────┐
│                    이미지 제작 8단계                             │
├─────────────────────────────────────────────────────────────────┤
│                                                                  │
│  [1] 관련 문서 정합성 검토                                      │
│       ↓                                                         │
│  [2] 기존 이미지 재사용 검토                                    │
│       ※ 후보 2개 이상 → 사용자 확인 후 결정                    │
│       ↓                                                         │
│  [3] 코드로 생성 (차트/그래프) ┐                                │
│       ↓                        │                                │
│  [4] 웹 검색 (라이선스 확인)   │ ※ 생성 전 프롬프트 제시       │
│       ↓                        │    → 사용자 확인 후 생성       │
│  [5] 유니코드 다이어그램 또는  │                                │
│       생성 프롬프트 검토      │                                │
│       ↓                        │                                │
│  [6] SVG 직접 제작            ┘                                 │
│       ↓                                                         │
│  [7] 한글 표시 확인                                             │
│       ↓                                                         │
│  [8] 수정 vs 새로 만들기 비교 → 사용자 판단 요청               │
│                                                                  │
└─────────────────────────────────────────────────────────────────┘
```

### 3.3 단계별 상세

#### [1단계] 관련 문서 정합성 검토

| 검토 항목 | 설명 |
|----------|------|
| 문서 버전 | 대상 문서의 최신 버전 확인 |
| 용어 일치 | 문서에서 사용하는 용어와 이미지 내 텍스트 일치 |
| 수치 일치 | 범위, 단위, 예시 값이 문서와 동일 |
| 색상 일치 | 시뮬레이터/문서의 색상 테마와 통일 |
| 구조 일치 | 레이아웃, 패널 구성이 실제 화면과 동일 |

#### [2단계] 기존 이미지 재사용 검토

- 이전에 제작한 유사 이미지가 있는지 확인
- 수정하여 재사용 가능한지 판단
- 새로 만드는 것보다 효율적인 경우 재사용
- **재사용 후보가 2개 이상이면 사용자에게 제시하고 확인 후 결정**

> ⚠️ **3~6단계 공통 원칙**: 이미지를 새로 생성하기로 결정했다면, **해당 이미지의 구성 계획(프롬프트)을 먼저 사용자에게 제시**한 후 확인을 받고 생성합니다.

#### [3단계] 코드로 생성

| 도구 | 용도 |
|------|------|
| Python + Matplotlib | 정교한 그래프, 수학적 시각화 |
| JavaScript + Chart.js | 인터랙티브 차트 |
| D3.js | 복잡한 데이터 시각화 |

#### [4단계] 웹 검색

- 라이선스 확인 필수 (CC0, MIT 등)
- 상업적 사용 가능 여부 확인
- 출처 명시

#### [5단계] 유니코드 다이어그램 또는 생성 프롬프트 검토

SVG를 직접 제작하기 전에, **구조 확인을 위한 사전 검토물**을 사용자에게 제공합니다. 이미지의 성격에 따라 **방법 A** 또는 **방법 B** 중 적합한 것을 선택합니다.

**방법 A: 유니코드 다이어그램** (구조적 다이어그램에 적합)

| 항목 | 설명 |
|------|------|
| 적합 대상 | 플로우차트, 계층도, 비교표, 프로세스 등 **구조 중심** 이미지 |
| 형식 | 유니코드 박스문자(┌─┐│└┘), 화살표(→↓←↑), 기호(●○◆▶) 등 활용 |
| 포함 내용 | 주요 요소 배치, 텍스트 레이블, 연결 관계, 대략적 비율 |
| 확인 사항 | 사용자가 구조·배치를 승인한 후 SVG 제작 진행 |

**예시:**
```
┌─────────────────────────────┐
│         제목 영역            │
├──────────┬──────────────────┤
│  패널 A  │     패널 B       │
│  ● 항목1 │  ┌──────┐       │
│  ● 항목2 │  │ 차트  │       │
│          │  └──────┘       │
├──────────┴──────────────────┤
│         하단 설명            │
└─────────────────────────────┘
```

**방법 B: 이미지 생성 프롬프트** (일러스트·인포그래픽에 적합)

| 항목 | 설명 |
|------|------|
| 적합 대상 | 일러스트, 비유·시나리오 그림, 사진 기반 인포그래픽 등 **시각 표현 중심** 이미지 |
| 형식 | 구조화된 텍스트 프롬프트 (외부 이미지 생성 AI 또는 SVG 제작 지시용) |
| 필수 포함 | ①구성 요소 배치, ②각 요소의 내용·텍스트, ③스타일·색상, ④비율·해상도 |
| 확인 사항 | 사용자가 프롬프트를 승인한 후 이미지 생성 진행 |

**프롬프트 작성 형식:**
```
[이미지명] 그림 XX: 제목

■ 레이아웃: (가로형/세로형/정사각형, 섹션 구분)
■ 구성 요소:
  (1) 영역명 — 배경색, 포함 내용, 텍스트
  (2) 영역명 — 배경색, 포함 내용, 텍스트
  ...
■ 스타일: (일러스트/다이어그램/인포그래픽, 톤, 색상 팔레트)
■ 텍스트: (언어, 폰트 스타일)
■ 해상도: (너비×높이 px)
```

> 💡 **선택 기준**: 유니코드 박스로 구조를 충분히 표현 가능하면 **방법 A**, 일러스트·캐릭터·복잡한 시각 요소가 핵심이면 **방법 B**를 사용합니다. 두 방법을 **병행**(유니코드로 배치 + 프롬프트로 시각 스타일)하는 것도 권장합니다.

#### [6단계] SVG 직접 제작

SVG 기술 가이드(폰트, 크기, 레이아웃)는 **코딩관리지침** §14~15를 참조합니다.

#### [7단계] 한글 표시 확인

| 확인 항목 | 확인 방법 |
|----------|----------|
| font-family 값 | SVG 내 모든 텍스트가 `Noto Sans CJK KR`을 사용하는가 |
| 브라우저 렌더링 | SVG를 브라우저에서 열어 한글·이모지 정상 표시 확인 |
| 텍스트 잘림/겹침 | 폰트 메트릭 차이로 위치 이탈이 없는가 |
| **PDF 변환 검증** | `rsvg-convert -f pdf`로 변환 후 한글 깨짐·□ 없는지 확인 |

> ⚠️ `Noto Sans KR`(Google 웹폰트)은 브라우저에서는 정상이지만 **rsvg-convert PDF 변환 시 한글이 깨집니다**. 반드시 시스템 폰트인 `Noto Sans CJK KR`을 사용하세요. PDF 변환 상세 절차: **문서작성지침서** §3.11 참조

#### [8단계] 수정 vs 새로 만들기 비교

- 두 옵션의 장단점 제시
- 예상 소요 시간 안내
- 최종 결정은 사용자에게 위임

## 4. 이미지 파일명 규칙

### 4.1 기본 형식

```
[문서종류]_[번호]_[내용]_v[버전]_[수정키워드]_[날짜]_[시간].svg
```

| 구성요소 | 설명 | 예시 |
|----------|------|------|
| 문서종류 | 이미지가 삽입될 문서 | 사용설명서, 이론강의서, 활용가이드, 수업지도안 |
| 번호 | 문서 내 순서 | 01, 02, 03 (2자리) |
| 내용 | 이미지 내용 요약 | 전체레이아웃, 회귀식요소 |
| 버전 | vX.Y.Z.W | v1.0.0.0, v1.0.0.2 |
| 수정키워드 | 해당 버전의 주요 변경 | 화살표제거, 간격조정 |
| 날짜 | YYYY-MM-DD | 2026-02-01 |
| 시간 | HHMM (24시간) | 0945 |

### 4.2 예시

```
✅ 사용설명서_04_모드전환_탭_v1.0.0.2_화살표제거_2026-02-01_0935.svg
✅ 이론강의서_02_단순선형회귀_v1.0.0.9_델타간격2px_2026-02-01_0340.svg
✅ 활용가이드_01_수업흐름_1차시_v1.0.0.2_원아래이동_2026-02-01_0200.svg
```

### 4.3 원본 파일

최초 생성 시 버전 표기 생략 가능:
```
사용설명서_01_추론과정.svg  ← 원본 (v1.0.0.0과 동일)
```

## 6. 이미지 수정 지시 해석

### 6.1 빨간 표시 규칙

사용자가 이미지에 **빨간색 원, 화살표, 표시**를 추가하여 업로드하면 **수정 지시**로 해석합니다.

| 표시 유형 | 해석 |
|----------|------|
| 빨간 원 | 해당 요소 수정 필요 |
| 빨간 화살표 | 이동 방향/위치 지시 |
| 빨간 X | 해당 요소 삭제 |
| 빨간 텍스트 | 변경할 내용 |
| 빨간 밑줄 | 강조/수정 대상 |

### 6.2 수정 지시 파악 절차

1. **기존 SVG와 비교**: 원본과 표시된 이미지 대조
2. **표시 요소 식별**: 빨간 표시가 가리키는 대상 파악
3. **수정 내용 추론**: 이동 위치, 추가/삭제 요소, 레이블 변경, 크기/간격 조정
4. **확인 요청**: 불명확한 경우 사용자에게 확인

### 6.3 예시

```
[사용자가 빨간 화살표로 텍스트 위치를 아래로 가리킴]
→ 해당 텍스트를 화살표 방향으로 이동

[사용자가 요소에 빨간 X 표시]
→ 해당 요소 삭제

[사용자가 빨간 원으로 두 요소 사이를 표시]
→ 간격 조정 필요
```

## 7. 문서 삽입 규칙

### 7.1 비율 유지 (Aspect Ratio)

문서에 이미지 삽입 시 **가로/세로 비율을 항상 동일하게 유지**합니다.

| 규칙 | 설명 |
|------|------|
| 비율 고정 | 확대/축소 시 가로·세로 비율 동일 |
| 왜곡 금지 | 한쪽만 늘이거나 줄이지 않음 |
| 원본 비율 | viewBox 비율 그대로 유지 |

### 7.2 고해상도 출력

| 규칙 | 설명 |
|------|------|
| SVG 우선 | 벡터 형식으로 해상도 무관 |
| PNG 사용 시 | 최소 2x 해상도 (Retina 대응) |
| 압축 최소화 | 품질 손실 없는 형식 사용 |

### 7.3 권장 삽입 크기

| 문서 유형 | 권장 너비 | 비고 |
|----------|----------|------|
| Word/HWP | 15cm (본문 너비) | 페이지 여백 고려 |
| PDF | 100% viewBox | 원본 크기 |
| 웹 문서 | max-width: 100% | 반응형 |
| PPT | 슬라이드 너비 90% | 여백 확보 |

## 10. 검토 체크리스트

### 10.1 프론트엔드-백엔드 연동

| # | 항목 | 확인 방법 |
|:-:|------|----------|
| 1 | API 엔드포인트 일치 | JS fetch URL ↔ Python @app.route |
| 2 | 요청/응답 형식 일치 | JS에서 기대하는 필드 ↔ Python 반환 필드 |
| 3 | HTTP 메서드 일치 | GET/POST 등 |
| 4 | 에러 응답 처리 | 실패 시 JS에서 처리 여부 |

```bash
# API 일치 확인 명령어
echo "=== JS 호출 ===" && grep -o "fetch('/api/[^']*" 파일.py | sort | uniq
echo "=== Python 라우트 ===" && grep -o "@app.route('/api/[^']*" 파일.py | sort | uniq
```

### 10.2 HTML/JavaScript

| # | 항목 | 확인 방법 |
|:-:|------|----------|
| 1 | ID 참조 일치 | JS에서 사용하는 ID ↔ HTML에 정의된 ID |
| 2 | 이벤트 핸들러 연결 | onclick, addEventListener 등 |
| 3 | 함수 정의 존재 | 호출하는 함수가 실제로 정의되어 있는가 |
| 4 | 괄호 균형 | ( ), { }, [ ] 쌍 맞음 |

```bash
# ID 참조 확인
grep -o '\$([^)]*)\.' 파일.py | sort | uniq  # JS 참조
grep -o 'id="[^"]*"' 파일.py | sort | uniq   # HTML 정의

# 괄호 균형 확인
echo "( ): $(grep -o '(' 파일 | wc -l) vs $(grep -o ')' 파일 | wc -l)"
echo "{ }: $(grep -o '{' 파일 | wc -l) vs $(grep -o '}' 파일 | wc -l)"
```

### 10.3 UI 상태 관리

| # | 항목 | 확인 방법 |
|:-:|------|----------|
| 1 | 버튼 활성화/비활성화 | 모든 상태 전환 시점에서 일관성 |

| 4 | 로딩 상태 표시 | 비동기 작업 중 피드백 |

### 10.4 하드코딩 검출

| # | 유형 | 예시 | 해결 |
|:-:|------|------|------|
| 1 | 개발자 경로 | `G:\내 드라이브` | 빈 값 또는 상수 |

## 11. 버그 분류와 영향 분석

### 11.1 심각도별 분류

#### 🔴 치명적 (Critical)

| 유형 | 증상 | 예시 |
|------|------|------|
| API 불일치 | 404 에러, 데이터 안 옴 | `/api/duplicates` vs `/api/results` |
| 참조 오류 | TypeError, null 접근 | `$('saveBtn')` (존재하지 않음) |
| 무한 루프 | 브라우저 멈춤 | while 조건 오류 |
| 데이터 손실 | 저장 안 됨, 삭제 오류 | 파일 경로 오류 |

#### 🟠 중간 (Medium)

| 유형 | 증상 | 예시 |
|------|------|------|
| UI 상태 불일치 | 버튼 활성화 오류 | 빈 결과인데 삭제 버튼 활성 |
| 표시 오류 | 값이 안 보임 | statSize 미업데이트 |
| 기능 누락 | 일부 기능 동작 안 함 | fmtSize 함수 없음 |
| 부분 실패 | 특정 조건에서만 오류 | 빈 그룹 미제거 |

#### 🟡 경미 (Low)

| 유형 | 증상 | 예시 |
|------|------|------|
| 데드 코드 | 없어도 동작함 | 미사용 함수/import |
| 하드코딩 | 다른 환경에서 문제 | 개발자 경로 |
| 비효율 | 느리거나 중복 | 중복 import |
| 스타일 | 코드 품질 저하 | 네이밍 불일치 |

### 11.2 영향 분석

#### 영향 분석 체크리스트

| # | 질문 | 확인 방법 |
|:-:|------|----------|
| 1 | 이 함수를 호출하는 곳은? | `grep -n "함수명(" 파일` |
| 2 | 이 변수를 참조하는 곳은? | `grep -n "변수명" 파일` |
| 3 | 이 API를 사용하는 곳은? | JS fetch 검색 |

| 5 | 수정 후 반환값/동작이 바뀌나? | 호출처 로직 확인 |

## 13. 수정 및 문서화

### 13.1 수정 절차

```
1. 버전 번호 증가 (패치: W+1)
2. 모든 버전 표시 위치 수정
   - docstring
   - HTML title
   - 헤더/푸터
   - 저장 파일 버전
   - 콘솔 출력
3. 버그 수정
4. 검증
5. 배치 파일 수정 (필요시)
```
