# Part 1: 리팩토링 패턴

> 이 Part는 기존 시뮬레이터의 구조를 변경하거나 규모를 확장할 때 적용하는 범용 패턴이다.
> ML확장리팩토링 지침서(v1.0.2.0)의 범용 패턴을 흡수하여 일반화하였다.
> ML 특화 패턴(데이터셋 객체 표준, 분류 확장 필드, Python 변환 스크립트)은 L3-ML로 이동되었다.

## 1. 개요

### 1.1 목적

기존에 동작하는 시뮬레이터의 **구조를 바꾸거나 규모를 확장**할 때, 안전하게 리팩토링하는 절차와 패턴을 정의한다.

### 1.2 적용 시점

| 상황 | 참조 문서 |
|------|----------|
| 새 시뮬레이터를 처음 만든다 | L1→L2→L3 코딩 표준 |
| 기존 시뮬레이터의 **구조를 바꾸거나 규모를 확장**한다 | **이 Part** |
| 데이터셋을 대폭 추가한다 | **이 Part** §3~4 |
| 버튼 UI를 드롭다운으로 전환한다 | **이 Part** §5 |
| 탭 내부 UI를 공유 영역으로 추출한다 | **이 Part** §6 |
| 함수명을 바꾸면서 기존 호출을 유지한다 | **이 Part** §9 |
| 코드 개선(교육 기능 강화, 시각화 추가) | Part 2 (§13) |
| SVG 제작·수정 | Part 2 (§14~15) |
| 개발일지·퀵가이드·수업 자료 작성 | Part 3 (§16~19) |

### 1.3 관련 문서

```
┌─────────────────────────────────────────────────────────────────┐
│                                                                  │
│  L1→L2→L3 코딩 표준        이 문서 Part 1       이 문서 Part 2  │

```

## 3. 데이터 구조 통합 패턴

### 3.1 문제 인식

시뮬레이터가 성장하면서 데이터 소스가 여러 객체로 분산되는 현상이 발생한다.

```
[전형적 성장 경로]
v1.0  초기 항목 4개
v1.1  + 업로드 기능
v1.2  + 내장 데이터 4개
v1.4  + 외부 데이터 23개 ← 여기서 구조 문제 폭발 (3중 구조?)
```

### 3.2 해법: 동질 통합 + 이질 분리

| 판단 기준 | 조치 |
|----------|:----:|
| 두 객체가 동일한 형태(구조)를 갖는다 | **통합** |
| 데이터 제공 방식이 근본적으로 다르다 (생성 vs 내장) | **분리** |
| 하나의 객체에 생성 파라미터와 실데이터가 혼재한다 | **분리 후 통합** |

### 3.3 적용 절차

```
[1단계] 기존 데이터 객체 전수 조사
[2단계] 동질/이질 판단
[3단계] 통합 객체 설계
[4단계] 로드 함수 단일화
[5단계] 기존 함수 래퍼 유지 (§9 참조)
```

### 3.4 적용 전 → 후

```javascript
// ❌ 적용 전: 2개 객체, 2개 로드 경로
const sourceA = { item1: { fields: [...], data: [...] } };
const sourceB = { item2: { fields: [...], data: [...] } };
function loadFromA(key) { /* sourceA에서 로드 */ }
function loadFromB(key) { /* sourceB에서 로드 */ }

// ✅ 적용 후: 역할별 분리, 동질 데이터 통합
const ITEMS = {
    item1: { name: '...', fields: [...], data: [...] },
    item2: { name: '...', fields: [...], data: [...] },
};
const GENERATORS = {
    random1: { range: [0,100], noise: 5 },
};
function loadSelectedItem() { /* ITEMS에서 로드 (단일 경로) */ }
function generateRandom() { /* GENERATORS에서 생성 (별도 경로) */ }
```

## 4. 파일 크기 관리 전략

### 4.1 행 압축 포맷

데이터를 배열로 임베딩할 때, 한 줄에 여러 행을 배치하여 줄 수를 줄인다.

| 데이터 행 수 | 줄당 행 수 | 예시 |
|:-----------:|:---------:|------|
| ≤ 30 | 3행/줄 | 소규모 데이터 |
| 31~100 | 5행/줄 | 중규모 데이터 |
| > 100 | 외부 파일 분리 | 대규모 데이터 |

### 4.2 키 단축

```javascript
// ❌ 긴 키 → 파일 비대
const ITEMS = { item: { sepal_length: [...], sepal_width: [...] } };

// ✅ 짧은 내부 키 + 레이블 매핑
const ITEMS = {
    item: {
        headers: ['c0', 'c1', 'c2'],
        headerLabels: ['꽃받침 길이(cm)', '꽃받침 너비(cm)', '꽃잎 길이(cm)'],
        data: [[5.1, 3.5, 1.4], [4.9, 3.0, 1.4]],
    }
};
```

### 4.3 카테고리는 HTML에서 관리

```javascript
// ❌ JS 데이터에 category 필드 → 드롭다운 생성 시 복잡
const ITEMS = { iris: { category: '자연과학', ... } };

// ✅ 카테고리는 HTML optgroup으로 직접 표현
// <optgroup label="🌿 자연과학">
//     <option value="iris">🌸 붓꽃 - 150개</option>
// </optgroup>
```

> **이유**: 카테고리는 UI 표시 관심사이므로 HTML에서 관리. JS 데이터 객체는 순수 데이터만 보유.

## 5. 셀렉터 UI 확장 패턴

### 5.1 전환 기준

| 항목 수 | 적합한 UI | HTML 구조 | 전환 시점 |
|:-------:|----------|----------|----------|
| 2~6개 | 버튼 그리드 | `<button>` | 초기 개발 |
| 7~15개 | 드롭다운 | `<select>` + `<option>` | 중간 확장 |
| 16개 이상 | 카테고리 드롭다운 | `<select>` + `<optgroup>` | 대규모 확장 |

### 5.2 전환 시 변경 범위

```
┌──────────────────────────────────────────────┐
│  변경 영역            작업 유형              │
├──────────────────────────────────────────────┤
│  CSS: 기존 버튼 스타일 제거, 드롭다운 스타일 추가  │
│  HTML: 버튼 그리드 → select+optgroup            │
│  JS: 로드 함수 통합, active 토글 코드 제거       │
└──────────────────────────────────────────────┘
```

### 5.3 카테고리 드롭다운 구현 템플릿

```html
<select id="itemSelect" class="item-selector" onchange="onItemChange()">
    <optgroup label="🌿 카테고리 A">
        <option value="item1">🌸 항목1 - 150개</option>
    </optgroup>
    <option disabled>──────────</option>
    <option value="_random">🎲 랜덤 생성</option>
</select>

<div class="item-info-box" id="itemInfoBox">
    <div id="itemDesc"></div>
    <div id="itemRec"></div>
</div>

<button class="btn btn-primary" onclick="loadSelectedItem()">📊 로드</button>
```

### 5.4 option 표기 규칙

```
<option value="키">이모지 한글명(영문명) - N개</option>
```

| 구성 요소 | 필수 | 예시 |
|----------|:----:|------|
| 이모지 | ✅ | 🌸, 🐧, 💼 |
| 한글명 | ✅ | 붓꽃, 펭귄 |
| 영문명 | ✅ | (Iris), (Penguins) |
| 행 수 | ✅ | - 150개 |


### 6.3 추출 절차 (4단계)

| 단계 | 작업 | 확인 사항 |

## 7. 조건부 UI 토글 패턴

### 7.1 패턴: classList.toggle

```javascript
function onItemChange() {
    const key = document.getElementById('itemSelect').value;
    const isSpecial = (key === '_random');

    // toggle(className, condition) — 한 줄로 표시/숨김 전환
    document.getElementById('itemInfoBox').classList.toggle('hidden', isSpecial);
    document.getElementById('specialControls').classList.toggle('hidden', !isSpecial);
}
```

### 7.2 toggle vs if-else

```javascript
// ❌ 장황한 if-else
if (isSpecial) {
    document.getElementById('itemInfoBox').classList.add('hidden');
    document.getElementById('specialControls').classList.remove('hidden');
} else {
    document.getElementById('itemInfoBox').classList.remove('hidden');
    document.getElementById('specialControls').classList.add('hidden');
}

// ✅ toggle(class, condition)
document.getElementById('itemInfoBox').classList.toggle('hidden', isSpecial);
document.getElementById('specialControls').classList.toggle('hidden', !isSpecial);
```

### 7.3 적용 한계

| 토글 대상 수 | 적합한 방식 |
|:-----------:|-----------|
| 2개 | `classList.toggle` |
| 3개 이상 | 상태 객체 + 루프 |

## 8. 선택-실행 분리 패턴

### 8.1 2단계 전환

```
[1단계 (소규모): 선택 = 즉시 실행]
버튼 클릭 → 즉시 데이터 로드 + 시각화 갱신

[2단계 (대규모): 선택 = 미리보기, 실행 = 별도]
드롭다운 선택 → 정보 표시 (미리보기) → "로드" 버튼 → 데이터 로드 + 시각화 갱신
```

### 8.2 전환 판단 기준

| 조건 | 선택 방식 |
|------|:---------:|
| 항목 ≤ 6개, 모두 친숙 | 1단계 (즉시 실행) |
| 항목 > 6개, 또는 메타 정보가 중요 | 2단계 (미리보기 → 실행) |

## 9. 하위 호환 래퍼 패턴

### 9.1 원칙

```
구 함수명은 래퍼로 유지하되, 내부는 신규 함수로 위임한다.
```

### 9.2 래퍼 작성 템플릿

```javascript
// ── 신규 함수 (실제 로직) ──
function loadSelectedItem() { /* ... 전체 구현 ... */ }

// ── 호환 래퍼: 구 함수 → 신규 함수 위임 ──
function loadItem(key) {
    if (ITEMS[key]) {
        document.getElementById('itemSelect').value = key;
        onItemChange();
        loadSelectedItem();
    }
}
```

### 9.3 래퍼 필요 여부 판단

```bash
# 기존 함수의 모든 호출처 확인 (함수 정의 제외)
grep -n "loadItem\b" 파일.html | grep -v "function loadItem"
```

| 결과 | 조치 |
|------|------|
| HTML `onclick`에서 호출됨 | 래퍼 필수 |
| 다른 JS 함수에서 호출됨 | 래퍼 또는 호출처 수정 |
| 호출처 0건 | 래퍼 불필요, 삭제 |

## 10. 안전한 코드 제거 절차

### 10.1 3종류의 제거 대상

| 유형 | 예시 | 위험도 |
|------|------|:------:|
| 전역 변수 | `let currentItem = 'default';` | 중 |
| 함수 | `function loadItem(key) { ... }` | 높 |
| CSS 클래스 | `.item-btn { ... }` | 낮 |

### 10.2 안전 제거 절차

```
[1단계] 제거 대상의 모든 참조 검색
        grep -n "currentItem" 파일.html

[2단계] 참조가 0건 → 안전하게 제거 가능

[3단계] 참조가 N건 → 각 참조의 맥락 확인
        └─ 해당 코드도 함께 리팩토링 필요

[4단계] 제거 후 재검색으로 완전 제거 확인
        grep -c "currentItem" 파일.html  # 기대: 0
```

### 10.3 DOM 요소 조건부 접근

UI 구조 변경으로 특정 DOM 요소가 항상 존재하지 않게 되면, null 체크를 선행한다.

```javascript
// ❌ 항상 존재한다고 가정
document.getElementById('slider').addEventListener('input', (e) => { ... });

// ✅ null 체크 선행
const slider = document.getElementById('slider');
if (slider) {
    slider.addEventListener('input', (e) => { ... });
}
```

## 11. 부수 작업 체크리스트

대규모 리팩토링 후 **메인 작업은 완료했지만 놓치기 쉬운** 부수 작업 목록이다.

### 11.1 필수 부수 작업 6가지

| # | 부수 작업 | 적용 조건 | 미수행 시 증상 |
|:-:|----------|----------|---------------|
| 1 | `resetAll()`에 신규 UI 숨김/초기화 추가 | UI 요소를 추가하거나 이동한 경우 | 초기화 후 UI 잔상 |
| 2 | 초기화 함수에 null 체크 추가 | DOM 요소 존재 조건이 변경된 경우 | 페이지 로드 시 에러 |
| 3 | `DOMContentLoaded`에 초기화 호출 추가 | 새 컨트롤이 초기 상태를 필요로 하는 경우 | 초기 상태 미설정 |
| 4 | 구 변수/함수의 참조 검색 후 제거 | 함수/객체를 리팩토링한 경우 | 데드 코드 잔존 |
| 5 | 제거된 CSS 클래스의 HTML 참조 검색 | CSS 클래스를 삭제/이름 변경한 경우 | 스타일 미적용 |
| 6 | 버전 배지 업데이트 | 모든 변경 | 버전 불일치 |

### 11.2 수행 순서

```
[메인 리팩토링 완료]
  → [1] resetAll() 점검
  → [2] 초기화 함수 null 체크
  → [3] DOMContentLoaded 점검
  → [4] 데드 코드 검색
  → [5] CSS-HTML 정합성
  → [6] 버전 배지
```

| 요소 | 간격 |

| 요소 간 최소 간격 | **10px** |

- **사전 계산**: 배치 전 좌표 계산으로 겹침 확인

## 17. 퀵가이드 작성법

### 17.1 퀵가이드의 정체성

퀵가이드는 **소프트웨어 옆에 띄워두는 '내비게이션'**입니다.

| 문서 유형 | 목적 | 사용 시점 | 깊이 |
|----------|------|----------|:----:|
| **퀵가이드** | 즉시 실행, 빠른 참조 | 소프트웨어 사용 **중** | ⭐ |
| 사용설명서 | 전체 기능 상세 설명 | 소프트웨어 사용 **전/후** | ⭐⭐⭐ |
| 이론해설서 | 개념 및 원리 학습 | 별도 학습 시간 | ⭐⭐⭐⭐ |
| 활용가이드 | 수업 적용 방법 | 수업 준비 시 | ⭐⭐⭐ |

> **핵심 질문**: "사용자가 소프트웨어 화면과 이 문서를 동시에 보고 있다면, 이 내용이 필요한가?"

### 17.2 4대 설계 원칙

| 원칙 | 설명 |
|------|------|
| **Action First** | 설명보다 행동 지시가 먼저. `❌ "선형 회귀는..."` → `✅ "[데이터 생성] 버튼을 클릭하세요"` |
| **Success First** | 30초 내 성공 경험 제공. `실습 → 성공! → "왜?" → 이론` 순서 |
| **Observe, Don't Just Do** | 관찰 유도. `❌ "실행 클릭"` → `✅ "실행 클릭 → 손실 그래프가 L자로 꺾이나요?"` |
| **Escape Route** | 탈출구 제공. `🆘 선이 춤춰요 → 학습률을 0.01 이하로 낮추세요` |

### 17.3 필수 섹션

| 순서 | 섹션명 | 목적 | 권장 분량 |
|:---:|--------|------|:--------:|
| 1 | **📖 이 문서 사용법** | 퀵가이드 활용 방법 안내 | 0.5쪽 |
| 2 | **⚡ 초속성 실습** | 30초~1분 내 성공 경험 | 0.5쪽 |
| 3 | **🚀 빠른 시작** | 핵심 기능 3단계 | 0.5쪽 |
| 4 | **🖥️ 화면 구성** | UI 레이아웃 안내 | 0.5쪽 |
| 5 | **🔍 관찰 포인트** | 무엇을 봐야 하는지 | 0.5쪽 |
| 6 | **🆘 트러블슈팅** | 문제 상황별 해결책 | 0.5쪽 |
| 7 | **🔑 핵심 용어** | 용어 정의 (학술+쉬운) | 0.5쪽 |

### 17.4 선택 섹션

| 섹션명 | 포함 조건 |
|--------|----------|

### 17.6 용어 처리 전략 (3층 구조)

모든 전문 용어는 **3가지 표현**을 준비합니다:

| 층위 | 용도 | 예시 |
|------|------|------|

| **번역 용어** | 한국어 대응 | 기울기, 학습률 |
| **쉬운 설명** | 직관적 이해 | "품질 1점 = 가격 몇 원?" |

```
❌ 학술 용어만: "Learning Rate를 조절하여 step size를 변경합니다."
❌ 쉬운 말만: "배우는 속도를 조절해서 한 걸음 크기를 바꿔요."
✅ 병기: "학습률(Learning Rate)을 조절하세요. → 한 번에 얼마나 배울지 결정합니다."
```

| **관찰 없는 행동** | "1. 생성 클릭 2. 실행 클릭" | "1. 생성 클릭 → 점이 나타났나요? 2. ..." |
| **탈출구 없음** | 트러블슈팅 미제공 | 증상-원인-해결책 테이블 |
| **과도한 친절** | 장문의 쉬운 설명 | 테이블 한 칸에 학술+쉬운 설명 병기 |
